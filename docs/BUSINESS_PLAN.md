# Polimoney Ledger - ビジネスプラン

## 1. サービス概要

政治団体・選挙運動の会計担当者向けの収支管理アプリケーション。
デジタル民主主義 2030（DD2030）がホスティングし、Polimoney と連携して政治資金の透明化を推進する。

---

## 2. ホスティング方針

### 2.1. 変更の背景

当初はセルフホスト（ユーザー各自が Supabase を用意）を想定していたが、以下の理由から DD2030 ホスティングに変更：

1. **ユーザビリティ**: 一般の会計担当者に Supabase セットアップは困難
2. **データの一貫性**: 選挙識別子・政治団体識別子を全ユーザーで共通化
3. **Polimoney 連携**: 共通 DB を参照することでリアルタイム連携が可能

### 2.2. セキュリティ方針

- **認証**: Supabase Auth（OTP 方式）
- **認可**: PostgreSQL RLS（Row Level Security）で論理的にデータ分離
- **暗号化**: At-rest & In-transit 暗号化
- **監査**: 全変更を監査ログに記録
- **インフラ**: Azure 上でホスティング（将来的には Supabase Self-Host も検討）

| プラン         | 対象               | アクティブ台帳 | 過去台帳 | ストレージ | メンバー | 年額       |
| -------------- | ------------------ | -------------- | -------- | ---------- | -------- | ---------- |
| **Free**       | 政治家 ID 発行     | 0              | 0        | -          | 1        | **¥0**     |

**金額について:**

- 寄付収入も見込めるため、料金は後日調整
- 過去台帳は資産として無料で保持（閲覧可能）

### 3.3. アクティブ台帳の仕様

**アクティブ/過去の切り替え:**

- ユーザーが手動で切り替え可能
- 過去台帳 → アクティブ: 上限以内なら可能
- アクティブ → 過去: いつでも可能

**過去台帳の修正:**

1. 一時的に「アクティブ」に切り替え
2. 修正を実施
3. 「過去」に戻す

**台帳数に余裕を持たせる理由:**

- 選挙プラン（2 台帳）: 後援会 + 選挙
- 議員プラン（4 台帳）: 後援会 + 選挙 + 修正用の余裕
- 政党プラン（10 台帳）: 複数団体 + 選挙 + 余裕

### 3.6. 年度の扱い

**政治団体台帳は年度ごとに増えない（1 台帳、年度フィルタ方式）:**

```
政治団体A（後援会）← 1台帳のまま
├── 2024年の仕訳（journal_date でフィルタ）
├── 2025年の仕訳
├── 2026年の仕訳
└── ...
```

- 収支報告書出力時に年度を指定してフィルタ
- 前年度繰越は自動計算
- 一般的な会計ソフト（弥生、freee）と同じ設計
- **10 年使っても政治団体台帳は 1 つのまま**

**選挙台帳は選挙ごとに作成:**

- 2023 年選挙 → 台帳 1
- 2027 年選挙 → 台帳 2（新規作成）
- 報告書提出後は「過去」に移行 → アクティブ枠を解放

### 3.4. 各プランの想定ユースケース

**Free（¥0）:**

- 政治家 ID を発行（永続的に保持）
- 台帳は作成不可
- 「いつか立候補するかも」という人のエントリーポイント

**選挙プラン:**

- 後援会台帳 + 選挙台帳
- 選挙終了後、台帳を「過去」に移行
- 落選者はこれで十分

**議員プラン:**

- 後援会台帳（継続利用）+ 選挙台帳（4 年周期）
- 過去の選挙台帳は無制限に保持
- 地方議員の標準プラン

**政党プラン:**

- 複数の政治団体を管理
- 国会議員、政党支部向け
